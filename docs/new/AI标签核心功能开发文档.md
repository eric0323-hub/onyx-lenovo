# AI 标签核心功能模块开发文档

## 1. 文档目的

本文档用于明确当前项目中“AI 标签”功能模块在 MVP 阶段需要实现的核心业务范围，供开发、产品、测试对齐交付目标。

本文档描述的是项目内的一个功能模块，不是一个完整独立项目。该模块依赖当前项目已有的知识库、文档导入、文档存储、用户权限、任务执行与结果展示等基础能力。

本文档只说明“需要实现什么”和“业务规则是什么”，不说明接口设计、数据库表结构、Prompt 编写、模型选型或任务调度等技术实现细节。

---

## 2. 模块定位与适用范围

AI 标签模块服务于知识进入平台后的结构化处理流程，目标是将批量导入的文本内容转化为可摘要、可分类、可维护、可追溯的知识资产。

本文档中的“文档”和“文章”均指已导入知识库并进入处理链路的文本内容，二者不代表两类不同的数据对象。

本模块不承担完整知识库项目的全部能力，只聚焦以下 6 个核心能力：

1. 批量文本文件的 AI 自动 Summary
2. 根据业务提示词自动生成标签体系
3. 基于当前标签体系进行批量打标
4. 批量打标过程中可选择优化标签体系
5. 标签体系健康自检与人工维护
6. 展示打标结果、标签体系版本、标签变更影响

本模块的上游是当前项目已有的文档导入与解析能力，下游是当前项目已有的知识库使用、检索、筛选、展示等能力。

## 3. 已确认业务约束

### 3.1 模块边界约束

1. AI 标签模块是现有项目中的业务功能模块，不是完整项目或独立产品。
2. 本模块不定义知识库创建、用户体系、权限体系、文件上传、全文检索等通用能力。
3. 本模块只处理文档入库后与 Summary、标签体系、打标、标签维护相关的业务流程。
4. 本模块输出的标签结果与 Summary 结果需要被当前项目的知识库能力消费。

### 3.2 标签体系结构约束

1. 全平台共用一套统一标签体系，不按知识库拆分多套标签树。
2. 标签体系固定为三级结构，不支持调整为两级或四级及以上。
3. 文章最终打标结果只允许绑定三级标签。
4. 一级标签和二级标签只作为分类层级与聚合展示使用，不作为文章最终绑定标签。
5. 新增标签必须以完整三级路径表达。
6. 每个标签必须包含标签说明，标签说明用于界定该标签的语义、适用范围和与相近标签的区别。
7. 如果新增内容涉及一级或二级层级变化，也必须同时形成可用于文章绑定的三级标签。
8. 标签体系生成后需要人工确认才能作为初始版本正式生效。
9. 批量打标必须优先基于当前已生效标签体系执行。

### 3.3 标签体系版本约束

1. 初始标签体系确认生效后，形成标签体系版本。
2. 每次标签体系发生正式变更，均需要生成新的标签体系版本。
3. 标签体系版本至少应记录版本号、变更原因、变更来源、变更时间和变更内容。
4. 批量打标任务启动时必须绑定当时的标签体系版本。
5. 批量打标任务执行过程中，不应被外部新版本标签体系中途影响。
6. 批量打标任务执行过程中产生的候选新增标签，需要逐条经过标签体系健康自检 Agent 判断。
7. 标签体系健康自检 Agent 只判断候选新增标签是否冗余、是否可新增，不决定是否生成新版本。
8. 单个批量打标任务内产生的多个自检通过标签，在任务完成后统一整理汇总，并生成一个新的标签体系版本。
9. 并发批量打标任务产生的新增标签，在各任务完成后基于最新已生效标签体系进行整理去重落版，再生成新版本。
10. 本文档中的“整理去重落版”指对任务新增标签建议进行汇总、去重和落版，不代表提供已有标签合并能力。
11. 文章标签结果需要记录其使用的标签体系版本。
12. 当文章标签结果所依赖的标签被删除或失效时，文章应回归未打标签池子，等待重新打标。

### 3.4 Summary 约束

1. Summary 为文档入库后的自动处理能力，不要求用户手动单独发起。
2. Summary 结果需要可查看，并可作为后续批量打标的数据来源之一。
3. Summary 结果允许人工编辑，编辑后的人工版本优先级高于 AI 自动生成版本。
4. Summary 被人工修改后，该文章原有标签结果应失效，文章回归未打标签池子，等待重新打标。

### 3.5 批量打标约束

1. 批量打标支持基于原文内容执行，也支持基于 Summary 执行。
2. 单篇文档可命中多个三级标签。
3. 未开启标签体系优化时，系统只基于当前任务绑定的标签体系版本输出推荐标签或未匹配结果。
4. 开启标签体系优化时，系统才允许输出候选新增标签。
5. “无法覆盖”“冗余”“优化强度”等判断口径由 LLM Prompt 和模型处理策略承载，本文档不展开具体 Prompt 规则。
6. Prompt 需要在开发和测试阶段通过业务样例人工打磨确认，验收时按确认后的 Prompt 样例验证。

### 3.6 多 Agent 协作约束

1. 为降低单个 LLM 处理节点的任务复杂度，本模块采用多 Agent 协作的业务分工。
2. 打标 Agent 负责基于当前标签体系版本对文章进行标签推荐，并识别当前体系无法覆盖的内容。
3. 标签体系健康自检 Agent 负责判断候选新增标签是否与已有标签冗余、是否符合三级体系、是否适合加入标签体系。
4. 打标 Agent 不负责最终决定标签体系是否新增、修改或删除。
5. 标签体系健康自检 Agent 不删除已有标签，不合并已有标签。
6. 标签体系健康自检 Agent 不决定标签体系版本是否更新，版本更新由系统在任务完成或人工维护完成后处理。
7. 启用标签体系优化前的强提醒确认，视为管理员授权系统在自检通过后将新标签自动应用到触发该标签的文章，并在任务完成后统一生成新标签体系版本。

### 3.7 标签体系维护约束

1. 管理员可以人工新增、修改或删除标签。
2. 管理员人工新增标签时，也必须经过标签体系健康自检 Agent 判断是否冗余。
3. 若人工新增标签与已有标签语义重复或高度相似，系统应提示复用已有标签，而不是直接新增。
4. 本模块不提供标签合并能力。
5. 已有标签即使语义接近，也不自动合并或删除；可通过修改标签说明、层级定义等方式澄清其适用边界。
6. 删除任一级别标签都属于高影响级联删除操作。
7. 删除一级标签时，其下所有二级标签、三级标签一并删除。
8. 删除二级标签时，其下所有三级标签一并删除。
9. 删除三级标签时，仅删除该三级标签本身。
10. 删除标签前，系统必须检查该标签及其下级标签是否已经应用在文章上。
11. 若被删除范围内的三级标签已经应用在文章上，删除前必须强提示管理员：受影响文章将失去有效三级标签，并回归未打标签池子。
12. 标签体系的新增、修改、删除行为均需要保留变更记录，并生成新的标签体系版本。

### 3.8 未打标签池约束

1. 文章因 Summary 修改、标签删除、标签失效或打标未匹配进入未打标签池子。
2. 进入未打标签池子的文章可由系统自动加入待打标队列。
3. 管理员也可以从未打标签池子中选择文章并批量触发重新打标。
4. 重新打标时应拉取数据库中最新已生效的标签体系版本。

## 4. 核心业务流程

### 4.1 总体流程

1. 用户批量导入文本文件。
2. 系统对导入文件进行解析，并自动生成 Summary。
3. 管理员基于业务提示词生成初始三级标签体系。
4. 管理员预览并确认标签体系，确认后形成标签体系初始版本。
5. 管理员选择目标文档发起批量打标任务。
6. 系统为批量打标任务绑定当前标签体系版本。
7. 管理员选择打标来源为原文或 Summary。
8. 管理员选择是否在本次打标过程中启用标签体系优化。
9. 若启用标签体系优化，系统在任务开始前展示强提醒，说明本次任务可能产生新增标签、自动应用到触发文章，并在任务完成后统一生成新标签体系版本。
10. 打标 Agent 基于任务绑定的标签体系版本输出文章三级标签。
11. 若当前标签体系无法覆盖文章内容，且已开启标签体系优化，打标 Agent 输出候选新增三级标签及完整三级路径。
12. 标签体系健康自检 Agent 对候选新增标签进行冗余判断。
13. 若候选新增标签与已有标签冗余，自检 Agent 输出建议复用的已有三级标签。
14. 若候选新增标签不冗余，系统先将该标签作为本任务新增标签结果，并将该新标签自动应用到触发该标签的文章。
15. 任务完成后，系统汇总本任务内全部新增标签结果，整理去重后生成新的标签体系版本。
16. 若存在并发任务新增标签，系统在任务完成后基于最新已生效版本重新整理本任务新增标签，再生成新版本。
17. 系统输出打标结果、标签体系版本变化、未匹配内容和受影响文章。
18. 若管理员人工维护标签体系，新增标签同样进入自检流程；删除标签按级联删除规则执行。

### 4.2 关键原则

1. Summary、标签体系版本、打标结果、标签体系变更需要形成前后衔接的完整链路。
2. 文章最终只绑定三级标签。
3. 标签推荐优先复用当前已生效标签体系，新增标签是例外处理，不是默认行为。
4. 标签体系优化必须经过“候选新增标签 -> 健康自检 -> 任务内应用或复用 -> 任务完成后统一落版”的流程。
5. 系统需要区分“任务绑定版本”“最新标签体系版本”和“文章标签结果版本”。
6. 涉及标签体系结构变化的操作，必须保留变更记录和影响提示。

## 5. 功能一：批量文本文件的 AI 自动 Summary

### 5.1 功能目标

在文本文件批量入库后，系统自动为每篇文档生成 Summary，供用户查看，并支持后续批量打标直接使用 Summary 作为输入内容。

### 5.2 功能定义

1. 支持对批量导入的文本文件逐篇生成 Summary。
2. Summary 生成应属于文档入库处理链路中的标准步骤。
3. 每篇文档应保留对应的 Summary 结果。
4. Summary 结果需要支持列表查看。
5. Summary 结果需要允许人工编辑，编辑后以人工版本为准。
6. Summary 修改后，需要触发文章回归未打标签池子。

### 5.3 输入

1. 已导入平台的文本文件。
2. 文档基础信息，包括文档标题、所属知识库、所属目录等。

### 5.4 输出

每篇文档至少需要产出以下结果：

1. 文档标题
2. Summary 内容
3. Summary 生成时间
4. Summary 状态
5. 是否为人工编辑版本
6. 当前标签状态

### 5.5 业务规则

1. Summary 自动执行，无需用户单独创建 Summary 任务。
2. Summary 完成后，文档才进入可用于打标的状态。
3. Summary 查看场景需支持按知识库、目录筛选。
4. 若 Summary 失败，需要保留失败状态和失败标识。
5. 人工编辑 Summary 后，后续若选择基于 Summary 打标，应使用最新保存版本。
6. Summary 被人工修改后，文章当前标签结果失效，文章进入未打标签状态。

### 5.6 用户可感知结果

1. 用户可以看到每篇文档是否已生成 Summary。
2. 用户可以查看 Summary 内容。
3. 用户可以修改 Summary 内容。
4. 用户可以看到 Summary 修改后文章已回归未打标签池子。
5. 用户在批量打标时可以明确选择是否基于 Summary 执行。

## 6. 功能二：根据业务提示词自动生成标签体系

### 6.1 功能目标

基于管理员输入的业务场景描述，自动生成一套统一的三级标签体系，作为后续批量打标的标准分类基础。

### 6.2 功能定义

1. 支持管理员输入业务提示词。
2. 支持系统基于业务提示词生成标签体系。
3. 生成结果必须为三级标签结构。
4. 标签体系应包含标签名称、层级关系和标签定义。
5. 标签说明为必填内容。
6. 标签体系生成后需支持预览。
7. 标签体系确认后正式生效，并形成初始标签体系版本。

### 6.3 输入

1. 管理员填写的业务提示词。
2. 业务范围描述、分类偏好等业务语义信息。

### 6.4 输出

标签体系结果至少应包含：

1. 一级标签
2. 二级标签
3. 三级标签
4. 每个标签的名称
5. 每个标签的说明或定义
6. 每个标签的适用范围
7. 标签体系版本号

### 6.5 业务规则

1. 标签体系必须固定为三级。
2. 标签体系在确认前属于待生效状态。
3. 未确认的标签体系不能用于正式批量打标。
4. 标签体系一旦确认，即作为当前平台统一标签体系使用。
5. 标签体系生成结果需支持查看层级关系。
6. 标签体系生成结果应尽量避免同级重复、语义重叠和层级错位。
7. 初始标签体系确认后，后续打标与标签维护均应基于该体系版本机制进行。
8. 标签说明缺失时，不允许确认标签体系生效。

### 6.6 用户可感知结果

1. 用户可以输入业务提示词发起标签体系生成。
2. 用户可以看到树形标签结构。
3. 用户可以查看每个标签的定义。
4. 用户可以确认或取消本次生成结果。
5. 用户可以清楚区分“预览结果”和“已生效结果”。
6. 用户可以看到当前标签体系版本。

## 7. 功能三：批量打标

### 7.1 功能目标

针对已完成解析和 Summary 处理的文档，基于当前任务绑定的标签体系版本批量生成三级标签结果。

### 7.2 功能定义

1. 支持选择多篇文档发起批量打标。
2. 支持按知识库、目录、关键词等方式筛选目标文档。
3. 支持选择打标来源为原文或 Summary。
4. 支持基于任务绑定的标签体系版本生成标签结果。
5. 支持单篇文档返回多个三级标签。
6. 支持查看每个推荐标签的置信度。
7. 支持识别“当前标签体系无法覆盖”的文档或内容。
8. 支持在开启标签体系优化时输出候选新增三级标签。
9. 支持将自检通过的新标签自动应用到触发该标签的文章。
10. 支持在任务完成后统一生成新的标签体系版本。

### 7.3 输入

1. 待打标文档集合。
2. 打标来源选择：原文或 Summary。
3. 任务绑定的标签体系版本。
4. 是否开启标签体系优化。

### 7.4 输出

每篇文档至少应返回以下信息：

1. 文档标题
2. 三级标签列表
3. 每个三级标签的完整层级路径
4. 每个标签的置信度
5. 未匹配内容或未覆盖原因
6. 候选新增标签，仅在开启标签体系优化时输出
7. 标签结果使用的标签体系版本
8. 若有任务内新增标签，需标记其待落版状态
9. 当前标签状态

### 7.5 业务规则

1. 仅已完成解析和 Summary 处理的文档可进入打标流程。
2. 若选择基于 Summary 打标，则应使用当前可用的 Summary 内容。
3. 打标 Agent 必须优先从任务绑定的标签体系版本中选择三级标签。
4. 普通打标结果只能绑定已存在的三级标签。
5. 未开启标签体系优化时，打标 Agent 不输出新增标签，只输出低置信度、未匹配或需处理状态。
6. 开启标签体系优化时，打标 Agent 可以输出候选新增三级标签，但候选新增标签必须进入标签体系健康自检流程。
7. 候选新增标签自检冗余时，系统复用已有三级标签。
8. 候选新增标签自检不冗余时，系统将该标签作为本任务新增标签结果，并将该标签应用到触发该标签的文章。
9. 本任务内新增标签只应用于触发该标签的文章，不参与本任务后续其他文章的打标候选。
10. 任务完成后，系统统一整理本任务内新增标签结果，并生成新的标签体系版本。
11. 后续新任务重新打标时，应拉取数据库中最新已生效标签体系版本。

### 7.6 用户可感知结果

1. 用户可以选择文档并发起任务。
2. 用户可以看到任务执行进度。
3. 用户可以逐篇查看三级标签结果。
4. 用户可以看到哪些内容没有被当前标签体系良好覆盖。
5. 用户可以看到本次任务使用的标签体系版本。
6. 用户可以明确区分“已有三级标签”“候选新增标签”“任务内新增标签”和“已落版标签”。

## 8. 功能四：批量打标过程中可选择优化标签体系

### 8.1 功能目标

在批量打标过程中，对于当前标签体系无法良好覆盖的文档内容，支持输出候选新增三级标签，并通过标签体系健康自检 Agent 判断是否冗余。若自检通过，系统先将新标签应用到触发文章，并在任务完成后统一整理生成新的标签体系版本。

### 8.2 功能定义

1. 批量打标时支持开启或关闭标签体系优化。
2. 开启后支持设置优化强度。
3. 优化强度的具体判定口径由 LLM Prompt 和模型处理策略承载。
4. 开启后在正式执行前必须进行二次确认。
5. 二次确认需要明确说明：本次任务可能产生新增标签、将新标签自动应用到触发文章，并在任务完成后统一生成新标签体系版本。
6. 系统需要识别未能有效归入现有标签体系的文档或内容。
7. 打标 Agent 需要输出候选新增三级标签、完整三级路径和来源依据。
8. 标签体系健康自检 Agent 需要判断候选新增标签是否与已有标签冗余。
9. 若候选新增标签冗余，自检 Agent 应输出建议复用的已有三级标签。
10. 若候选新增标签不冗余，系统应将其纳入本任务新增标签结果。
11. 任务完成后，系统应统一整理本任务新增标签结果，并生成新的标签体系版本。

### 8.3 多 Agent 职责边界

打标 Agent 负责：

1. 基于当前标签体系版本对文章进行打标。
2. 判断文章内容是否能被当前标签体系覆盖。
3. 在开启优化时提出候选新增三级标签。
4. 给出候选新增三级标签的完整路径、来源依据和适用文章。

标签体系健康自检 Agent 负责：

1. 判断候选新增标签是否与已有标签语义重复。
2. 判断候选新增标签是否只是已有标签的同义表达、细粒度重复或命名变体。
3. 判断候选新增标签是否符合三级标签体系。
4. 判断候选新增标签的建议挂载位置是否合理。
5. 输出“复用已有标签”“新增标签”“拒绝新增”或“需处理”的判断。
6. 不负责合并已有标签。
7. 不负责删除已有标签。
8. 不负责决定标签体系版本更新时间。

管理员负责：

1. 确认是否开启标签体系优化。
2. 维护标签体系。
3. 执行删除标签等高影响操作。

### 8.4 输入

1. 批量打标任务。
2. 任务绑定的标签体系版本。
3. 是否开启标签体系优化。
4. 优化强度配置。
5. 打标 Agent 输出的候选新增三级标签。
6. 候选新增标签对应的文章和来源依据。

### 8.5 输出

当存在候选新增标签时，至少应输出：

1. 候选新标签名称
2. 完整三级路径
3. 建议挂载位置
4. 建议来源依据
5. 建议置信度
6. 自检判断结果
7. 若冗余，建议复用的已有三级标签
8. 若不冗余，任务内新增标签结果
9. 关联文章数量和文章列表
10. 标签应用结果
11. 任务完成后的标签体系版本

### 8.6 业务规则

1. 标签体系优化为可选项，不开启时仅执行普通批量打标。
2. 开启标签体系优化时，开始执行前必须弹出强提醒确认。
3. 强提醒确认后，系统可在自检通过时将新增标签自动应用到触发文章。
4. 候选新增标签必须先经过标签体系健康自检 Agent 判断。
5. 自检判断为冗余时，不得直接新增标签，应复用已有三级标签。
6. 自检判断为不冗余时，系统先记录为任务内新增标签，不立即生成新版本。
7. 任务内新增标签应自动用于触发该标签的原文章。
8. 任务内新增标签不参与本任务后续其他文章的打标候选。
9. 任务完成后，系统对全部任务内新增标签进行整理去重，并统一生成新的标签体系版本。
10. 并发任务完成时，应基于最新已生效标签体系重新整理本任务新增标签，再生成新版本。
11. 自检判断为拒绝新增或需处理时，文章进入未匹配或未打标签状态。
12. 启用优化时，应保留未匹配文档或未匹配内容的收集结果。
13. 标签体系更新行为需要保留变更记录。

### 8.7 用户可感知结果

1. 用户可以决定本次打标是否启用标签体系优化。
2. 用户可以在开始前收到明确确认提醒。
3. 用户可以看到未匹配内容和候选新增标签。
4. 用户可以看到自检 Agent 对候选新增标签的冗余判断。
5. 用户可以看到系统复用了哪个已有三级标签。
6. 用户可以看到任务完成后生成的标签体系版本。
7. 用户可以看到新标签自动应用到了哪些文章。

## 9. 功能五：标签体系健康自检与人工维护

### 9.1 功能目标

支持管理员在标签体系生效后进行人工维护，同时通过标签体系健康自检 Agent 降低新增冗余标签的风险，并通过级联删除规则保证三级标签体系结构完整。

### 9.2 功能定义

1. 支持管理员查看当前已生效标签体系。
2. 支持管理员人工新增标签。
3. 支持管理员人工修改标签名称、说明或挂载关系。
4. 支持管理员人工删除标签。
5. 人工新增标签必须经过标签体系健康自检 Agent 判断。
6. 删除标签前必须检查该标签及其下级标签是否已被文章使用。
7. 删除已使用标签时必须展示影响提示和二次确认。
8. 标签体系维护结果需保留变更记录，并生成新的标签体系版本。
9. 本模块不支持标签合并。
10. 标签说明为新增和修改标签时的必填信息。

### 9.3 人工新增标签规则

1. 管理员新增标签时，需要填写标签名称、完整三级路径和标签说明。
2. 新增标签提交后，先进入自检流程。
3. 自检 Agent 需要判断该标签是否与已有标签重复、相似或可被已有标签覆盖。
4. 若自检结果为冗余，系统应提示复用的已有三级标签，并说明原因。
5. 若自检结果为不冗余，系统允许新增该三级标签，并生成新的标签体系版本。
6. 人工新增标签不得破坏三级标签体系结构。
7. 人工新增标签只更新标签体系，不自动影响已有文章。

### 9.4 人工修改标签规则

1. 管理员可以修改标签名称和标签说明。
2. 修改标签名称或说明时，应保留原标签的历史记录。
3. 若修改会造成与其他标签语义重叠，系统应提示潜在冗余风险。
4. 若修改挂载关系，需要保证标签体系仍为三级结构。
5. 修改后的标签体系需要生成新的标签体系版本。
6. 已有标签不因语义接近而自动合并。
7. 修改后标签说明不得为空。

### 9.5 人工删除标签规则

1. 删除任一级别标签都属于高影响操作。
2. 管理员删除标签前，系统必须检查该标签及其下级标签是否已应用在文章上。
3. 删除一级标签时，其下所有二级标签、三级标签一并删除。
4. 删除二级标签时，其下所有三级标签一并删除。
5. 删除三级标签时，仅删除该三级标签本身。
6. 任一级别标签删除后，该标签所在的有效标签路径不再存在。
7. 若被删除范围内的三级标签未应用在任何文章上，可进入普通删除确认。
8. 若被删除范围内的三级标签已应用在文章上，系统必须展示强提示。
9. 强提示至少应包含：删除范围、受影响三级标签数量、受影响文章数量、代表性文章列表、删除后的影响说明。
10. 删除后的影响说明必须明确：受影响文章将失去有效三级标签，并回归未打标签池子，等待重新打标。
11. 删除已应用标签必须由管理员二次确认，不允许静默删除。
12. 删除后，受影响文章应进入未打标签或待重新打标状态。
13. 删除后需要生成新的标签体系版本。

### 9.6 标签合并规则

1. 本模块不提供标签合并能力。
2. 已有标签即使存在语义接近、同词多义或层级释义差异，也应保留。
3. 对于容易误解的已有标签，优先通过修改标签说明或调整层级定义来澄清，而不是合并或删除。
4. 自检 Agent 判断新增标签冗余时，只建议复用已有标签，不执行合并。

### 9.7 输出

标签体系维护至少应输出：

1. 标签变更类型：新增、修改、删除
2. 变更前内容
3. 变更后内容
4. 自检结果
5. 冗余判断依据
6. 影响文章数量
7. 受影响三级标签列表
8. 标签体系版本变化
9. 变更生效时间
10. 标签说明

### 9.8 用户可感知结果

1. 用户可以维护当前标签体系。
2. 用户新增标签时可以看到是否存在冗余。
3. 用户可以看到系统建议复用的已有三级标签。
4. 用户删除标签时可以看到级联删除范围。
5. 用户可以看到该操作影响哪些文章。
6. 用户可以明确知道删除已使用标签会导致相关文章回归未打标签池子。
7. 用户可以查看标签体系变更记录和版本变化。

## 10. 功能六：展示打标结果、标签体系版本与变更影响

### 10.1 功能目标

让管理员能够清晰查看批量打标任务的执行进度、每篇文档的三级标签结果、候选新增标签、自检判断、标签体系版本变化以及标签删除影响。

### 10.2 功能定义

1. 支持展示批量打标任务的整体状态。
2. 支持展示任务处理进度。
3. 支持展示每篇文档的三级标签结果。
4. 支持展示成功、失败、未匹配、未打标签、待重新打标等状态。
5. 支持展示本次任务绑定的标签体系版本。
6. 支持在启用标签体系优化时同步展示候选新增标签、自检结果和任务完成后的标签体系版本。
7. 支持展示标签体系变更对文章的影响。
8. 支持展示未打标签池子及其重新打标入口。

### 10.3 展示维度

任务维度至少包括：

1. 任务状态
2. 文档总数
3. 已处理数量
4. 成功数量
5. 失败数量
6. 未匹配数量
7. 当前处理进度
8. 任务绑定标签体系版本
9. 任务完成后标签体系版本
10. 任务内新增标签数量

文档维度至少包括：

1. 文档标题
2. 三级标签
3. 标签完整层级路径
4. 置信度
5. 未匹配原因
6. 标签体系版本
7. 当前标签状态
8. 是否来自任务内新增标签

标签优化维度至少包括：

1. 候选新增标签
2. 完整三级路径
3. 来源文章
4. 来源依据
5. 自检判断
6. 复用的已有三级标签
7. 任务内新增标签结果
8. 任务完成后的标签体系版本
9. 自动应用文章列表

标签变更影响维度至少包括：

1. 变更标签
2. 变更类型
3. 级联删除范围
4. 影响三级标签数量
5. 影响文章数量
6. 影响文章列表
7. 回归未打标签池子的文章列表

未打标签池维度至少包括：

1. 文档标题
2. 进入未打标签池原因
3. 原标签体系版本
4. 当前最新标签体系版本
5. 是否可自动加入待打标队列
6. 是否可被管理员批量选择重新打标

### 10.4 业务规则

1. 打标结果展示应区分任务层、文档层、标签优化层和标签变更影响层。
2. 展示结果应区分任务绑定版本、最新标签体系版本和文章标签结果版本。
3. 若文档处理失败，应可识别失败状态。
4. 若任务尚未完成，应持续展示处理中状态和进度。
5. 若文章因 Summary 修改或标签删除导致原标签失效，应显示未打标签或待重新打标状态。
6. 若启用标签体系优化，应展示自动新增标签及自动应用结果。
7. 未打标签池中的文章重新打标时，应使用最新已生效标签体系版本。

### 10.5 用户可感知结果

1. 用户可以知道任务是否已开始、进行中、已完成或已取消。
2. 用户可以看到每篇文档具体绑定了哪些三级标签。
3. 用户可以看到本次任务使用了哪个标签体系版本。
4. 用户可以看到本次任务是否伴随标签体系优化及其结果。
5. 用户可以看到自检 Agent 对新增标签的判断。
6. 用户可以看到标签体系变更影响了哪些文章。
7. 用户可以看到哪些文章已回归未打标签池子。
8. 用户可以从未打标签池子中批量触发重新打标。

## 11. 状态要求

### 11.1 文档状态

文档处理链路至少应覆盖以下状态：

1. 待解析
2. 解析中
3. 解析成功
4. 解析失败
5. 待 Summary
6. Summary 中
7. Summary 成功
8. Summary 失败
9. 未打标签
10. 打标中
11. 打标成功
12. 未匹配
13. 待重新打标
14. 已打标签

### 11.2 任务状态

批量打标任务至少应覆盖以下状态：

1. 待执行
2. 执行中
3. 已完成
4. 已取消
5. 执行失败

### 11.3 标签优化状态

标签体系优化场景下，候选新增标签至少应覆盖以下状态：

1. 待自检
2. 自检中
3. 已复用已有标签
4. 任务内新增
5. 待统一落版
6. 已落版
7. 已拒绝新增
8. 需处理

### 11.4 标签状态

标签体系中的标签至少应覆盖以下状态：

1. 待生效
2. 已生效
3. 已修改
4. 已删除

### 11.5 标签体系版本状态

标签体系版本至少应覆盖以下状态：

1. 待生效
2. 已生效
3. 已被新版本替代

### 11.6 标签变更状态

标签体系变更记录至少应覆盖以下状态：

1. 待执行
2. 已执行
3. 已取消

### 11.7 未打标签池状态

未打标签池中的文章至少应覆盖以下状态：

1. 待入队
2. 已入队
3. 待管理员触发
4. 重新打标中
5. 重新打标完成

## 12. 角色与操作边界

### 12.1 管理员可执行操作

1. 输入业务提示词并生成标签体系。
2. 预览并确认初始标签体系生效。
3. 发起批量打标任务。
4. 选择打标来源为原文或 Summary。
5. 选择是否启用标签体系优化。
6. 查看打标结果、标签优化结果和标签体系版本变化。
7. 人工新增、修改、删除标签。
8. 查看标签删除的级联影响。
9. 对删除标签等高影响操作进行二次确认。
10. 从未打标签池中批量触发重新打标。

### 12.2 打标 Agent 操作边界

1. 可以基于任务绑定的标签体系版本输出三级标签。
2. 可以识别标签体系无法覆盖的文章内容。
3. 可以在开启优化时输出候选新增三级标签。
4. 不可以直接删除标签。
5. 不可以直接合并标签。

### 12.3 标签体系健康自检 Agent 操作边界

1. 可以判断候选新增标签是否冗余。
2. 可以建议复用已有三级标签。
3. 可以判断不冗余的候选三级标签可新增。
4. 可以提示标签修改或删除的潜在风险。
5. 不可以删除已有标签。
6. 不可以合并已有标签。
7. 不可以决定标签体系版本更新时间。

### 12.4 普通使用者感知范围

1. 查看文档 Summary。
2. 查看文档标签结果。
3. 在权限允许范围内使用最终已生效的标签和摘要结果。
4. 不参与标签体系生成、优化和维护确认。

## 13. 本次文档不包含的内容

以下内容不属于本文档范围：

1. 接口设计
2. 数据库表结构设计
3. 技术选型说明
4. 算法或模型实现方案
5. Prompt 编写细节
6. 页面视觉稿和交互稿细节
7. 任务调度、队列、缓存等技术实现方式
8. 完整知识库项目的业务设计
9. 标签合并能力
10. Prompt 具体内容

## 14. 开发交付判定口径

当以下结果均可被业务侧验证时，可认为本次核心能力范围已覆盖：

1. 批量导入的文本文件可自动生成 Summary，并可查看与编辑。
2. Summary 被人工修改后，文章可回归未打标签池子。
3. 管理员可基于业务提示词生成并确认一套三级标签体系。
4. 初始标签体系确认后可形成标签体系版本。
5. 管理员可选择文档、选择打标来源，并完成批量打标。
6. 批量打标任务可绑定标签体系版本。
7. 文章最终标签结果只绑定三级标签。
8. 打标结果优先复用当前任务绑定的标签体系版本。
9. 管理员可选择是否在打标中启用标签体系优化。
10. 启用优化时，系统可输出候选新增三级标签。
11. 候选新增标签可经过标签体系健康自检 Agent 判断是否冗余。
12. 冗余候选标签可复用已有三级标签。
13. 非冗余候选标签可作为任务内新增标签自动应用到触发文章，并在任务完成后统一落入新的标签体系版本。
14. 管理员人工新增标签时，自检 Agent 可检查是否冗余。
15. 管理员删除标签时，系统可按级联规则展示删除范围。
16. 删除已应用标签后，受影响文章可回归未打标签池子。
17. 未打标签池文章可自动加入待打标队列，或由管理员批量触发重新打标。
18. Prompt 可通过业务样例人工打磨确认，并用于开发测试验收。
19. 系统可清晰展示任务进度、文档结果、候选标签、自检结果、标签体系版本和变更影响。
