# AI 标签与企业 Taxonomy 知识治理模块开发文档

## 1. 文档目的

本文档用于明确当前项目中“AI 标签与企业 Taxonomy 知识治理”功能模块在 MVP 阶段需要实现的核心业务范围，供开发、产品、测试对齐交付目标。

本文档描述的是项目内的一个功能模块，不是一个完整独立项目。该模块依赖当前项目已有的知识库、文档导入、文档解析、文档存储、用户权限、任务执行、全文检索、筛选展示等基础能力。

本文档只说明“需要实现什么”和“业务规则是什么”，不说明接口设计、数据库表结构、Prompt 具体内容、模型选型或任务调度等技术实现细节。

---

## 2. 模块定位与适用范围

AI 标签模块的目标不是简单为文档生成若干松散标签，而是通过一套可版本化、可维护、可迭代的三级 Taxonomy，将企业知识文档结构化治理为可分类、可检索、可筛选、可追溯的知识资产。

本文档中的“文档”指已导入知识库并进入处理链路的文本内容。

本文档中的“标签体系”“Taxonomy”“分类体系”指同一类能力：由一级、二级、三级节点组成的企业知识分类树。文档最终只能绑定第三级叶子节点。

本模块聚焦以下 7 个核心能力：

1. 批量文本文件的 AI 自动 Summary。
2. 提供企业通用默认 Taxonomy 模板。
3. 支持管理员按三级节点多选启用默认分类。
4. 支持管理员基于行业、企业描述和知识库内容生成企业特色 Taxonomy 草稿。
5. 基于当前生效 Taxonomy 版本进行批量 AI 打标。
6. 支持 Taxonomy 健康自检、人工维护、版本变更和影响分析。
7. 将最终三级标签结果提供给知识库检索、筛选、展示和后续治理流程消费。

本模块的上游是当前项目已有的文档导入与解析能力，下游是当前项目已有的知识库使用、检索、筛选、展示等能力。

## 3. 核心概念

### 3.1 Taxonomy

Taxonomy 是企业知识文档的三级分类树。

示例：

```text
企业知识库 Taxonomy
├── 01. 公司治理与合规
│   ├── 政策与制度
│   │   ├── 公司章程
│   │   ├── 管理制度
│   │   └── 合规政策
│   ├── 风险与内控
│   └── 审计报告
├── 02. 人力资源文档
│   ├── HR 政策
│   │   ├── 员工手册
│   │   ├── 假期政策
│   │   └── 考勤制度
│   ├── 薪酬与福利
│   └── 招聘与离职
└── 03. 技术与研发
    ├── 技术规范
    ├── 项目文档
    └── 专利与知识产权
```

### 3.2 Taxonomy 节点

Taxonomy 节点至少包含：

1. 节点 ID。
2. 节点名称。
3. 所属层级：一级、二级或三级。
4. 父节点。
5. 节点定义。
6. 适用范围。
7. 正例。
8. 反例。
9. 关键词。
10. 同义词。
11. 节点来源：系统默认、行业模板、企业生成、人工新增。
12. 节点状态。
13. 所属 Taxonomy 版本。

三级节点还需要能够作为文档最终绑定标签。

以上字段可以支撑 MVP 的基础展示和打标，但如果要支撑企业知识治理、后续重分类、检索过滤、版本追溯和人工运营，仅作为最小字段仍然不够。补充字段见 3.7 和 3.8。

### 3.3 默认 Taxonomy 模板

默认 Taxonomy 模板是平台预置的企业通用分类体系，用于降低企业初始化成本。

默认模板应覆盖常见企业知识文档类型，例如：

1. 公司治理与合规。
2. 人力资源文档。
3. 财务与会计。
4. 法务与合同。
5. 行政与办公。
6. 销售与市场。
7. 客户服务与售后。
8. IT 与信息安全。
9. 技术与研发。
10. 项目管理。
11. 采购与供应链。
12. 质量、安全与审计。

默认模板只是候选模板，不应在未经过管理员确认时直接作为企业生效 Taxonomy。

### 3.4 行业与企业特色 Taxonomy

行业 Taxonomy 是面向特定行业的分类补充，例如重工机械行业可以包含：

1. 设备运维。
2. 工艺规程。
3. 质量检验。
4. 安全生产。
5. 备件管理。
6. 售后维修。
7. 产品手册。
8. 生产排程。

企业特色 Taxonomy 是根据企业自身知识库内容、业务描述、行业背景、组织结构和管理习惯生成或人工维护的分类补充。

行业与企业特色 Taxonomy 均应先进入草稿或候选状态，经管理员确认后才可生效。

### 3.5 叶子标签

叶子标签指 Taxonomy 的第三级节点。

文档最终打标结果只能绑定叶子标签。一级和二级节点只用于分类聚合、导航展示、批量选择和影响分析。

### 3.6 标签结果

标签结果是文档与一个或多个三级叶子标签之间的绑定关系。

标签结果至少需要记录：

1. 文档。
2. 三级叶子标签。
3. 完整三级路径。
4. 标签置信度。
5. 标签来源：AI 推荐、任务内新增、管理员确认、重新打标。
6. 使用的 Taxonomy 版本。
7. 当前有效状态。

### 3.7 Taxonomy 字段充分性审核结论

当前文档中的 Taxonomy 字段已经覆盖了节点语义描述的核心要素：名称、层级、父节点、定义、适用范围、正例、反例、关键词、同义词、来源、状态和版本。

这些字段足够支撑：

1. 初始 Taxonomy 草稿生成。
2. 管理员人工查看和编辑节点。
3. 基础批量 AI 打标。
4. 标签结果展示。
5. 基于三级叶子标签的检索筛选。

但在企业知识治理场景中，如果只保留这些字段，会在以下场景中不够用：

1. 节点改名后，历史标签结果难以稳定追溯。
2. 节点移动后，旧路径和新路径之间的影响范围难以判断。
3. 多个同义、近义、跨部门用语无法形成统一命名和别名管理。
4. 打标结果无法解释“为什么打了这个标签”。
5. 低置信度、待复核、人工确认和自动推荐之间的边界不清楚。
6. 删除、停用、替换节点时，无法指定推荐迁移去向。
7. 不同知识库、目录、文档类型或部门是否适用同一节点无法表达。
8. 后续想做 Taxonomy 健康度、覆盖率、使用率和命中率统计时缺少治理字段。

因此，本模块应将字段分为两层：

1. MVP 必填字段：保证 Taxonomy 可以被生成、确认、打标、展示和过滤。
2. 治理增强字段：保证 Taxonomy 后续可维护、可审计、可解释、可重分类、可运营。

### 3.8 Taxonomy 推荐字段分层

以下字段是业务字段建议，不代表具体数据库表结构。实现时可以拆分为多张表、JSON 字段、版本快照或只读统计视图。

#### 3.8.1 Taxonomy 体系字段

Taxonomy 体系表示某个企业或租户下的一套分类体系。

| 字段 | 必要性 | 说明 |
| --- | --- | --- |
| Taxonomy ID | 必填 | 稳定 ID，用于识别一套分类体系。 |
| 企业或租户 ID | 必填 | 标识该 Taxonomy 归属。 |
| Taxonomy 名称 | 必填 | 业务展示名称，例如“企业知识库 Taxonomy”。 |
| 适用范围 | 必填 | MVP 默认为企业或租户级；后续可扩展到知识库、部门、文档集。 |
| 当前生效版本 | 必填 | 指向当前已生效 Taxonomy 版本。 |
| 默认语言 | 建议 | 用于中英文、多语言别名和后续展示。 |
| 行业背景 | 建议 | 生成企业特色 Taxonomy 时的行业语境。 |
| 企业描述 | 建议 | 生成和解释 Taxonomy 的业务上下文。 |
| 管理负责人 | 建议 | 负责维护该 Taxonomy 的管理员或团队。 |
| 创建时间 | 必填 | 审计字段。 |
| 更新时间 | 必填 | 审计字段。 |

#### 3.8.2 Taxonomy 版本字段

Taxonomy 版本表示一次可追溯、可回看的正式分类体系快照。

| 字段 | 必要性 | 说明 |
| --- | --- | --- |
| Taxonomy 版本 ID | 必填 | 稳定版本 ID。 |
| Taxonomy ID | 必填 | 所属分类体系。 |
| 版本号 | 必填 | 例如 v1、v2、v2026.06.03。 |
| 版本状态 | 必填 | 草稿、待生效、已生效、已被新版本替代、已废弃。 |
| 父版本 ID | 建议 | 用于追溯从哪个版本演进而来。 |
| 版本来源 | 必填 | 初始生成、默认模板选择、人工维护、批量打标优化、重新落版。 |
| 变更摘要 | 必填 | 管理员可读的版本变化说明。 |
| 变更原因 | 建议 | 为什么要生成该版本。 |
| 生效时间 | 必填 | 判断打标任务应绑定哪个版本。 |
| 创建人 | 必填 | 发起或创建该版本的人。 |
| 确认人 | 建议 | 最终确认生效的人。 |
| 节点数量统计 | 建议 | 一级、二级、三级节点数量，便于健康度查看。 |
| 健康自检结果 | 建议 | 重复、缺定义、粒度不均、层级错位等自检摘要。 |

#### 3.8.3 Taxonomy 节点字段

Taxonomy 节点表示一级、二级或三级节点。三级节点是最终可绑定到文档的叶子标签。

| 字段 | 必要性 | 说明 |
| --- | --- | --- |
| 节点 ID | 必填 | 不随改名变化的稳定 ID。 |
| 节点编码 | 建议 | 人类可读的稳定编码，例如 `HR.POLICY.LEAVE`，便于导入导出和排查。 |
| 所属 Taxonomy 版本 | 必填 | 节点属于哪个版本快照。 |
| 父节点 ID | 必填 | 一级节点父节点为空。 |
| 所属层级 | 必填 | 一级、二级、三级。 |
| 节点名称 | 必填 | 管理员和用户看到的主名称。 |
| 展示名称 | 建议 | 用于界面展示，可与治理名称不同。 |
| 完整路径 | 必填 | 展示路径快照，例如“人力资源文档 / HR 政策 / 假期政策”。 |
| 路径 ID 列表 | 建议 | 结构化记录祖先节点，便于移动和影响分析。 |
| 排序值 | 建议 | 控制同级节点展示顺序。 |
| 节点定义 | 必填 | 说明该节点是什么。 |
| 适用范围 | 必填 | 说明适合哪些文档、知识库、部门、业务场景。 |
| 不适用范围 | 建议 | 说明不应命中的边界，降低误打标。 |
| 正例 | 必填 | 典型应命中文档或内容。 |
| 反例 | 必填 | 容易混淆但不应命中的文档或内容。 |
| 关键词 | 建议 | 用于生成、检索辅助和人工理解。 |
| 同义词 | 建议 | 覆盖企业内部叫法、简称和行业术语。 |
| 多语言名称 | 建议 | 中文名、英文名、别名等。 |
| 打标指导 | 建议 | 给打标 Agent 使用的判定口径，区别于面向人的说明。 |
| 冲突处理规则 | 建议 | 与相近节点同时命中时如何选择或是否允许同时保留。 |
| 是否可绑定文档 | 必填 | 仅三级叶子标签为是，一级和二级为否。 |
| 是否允许多标签共存 | 建议 | 用于判断该节点是否可以与其他叶子标签同时绑定。 |
| 最低自动打标置信度 | 建议 | 低于该阈值进入待确认或未匹配。 |
| 人工复核阈值 | 建议 | 位于区间内的标签进入低置信度待确认。 |
| 节点来源 | 必填 | 系统默认、行业模板、企业生成、人工新增、任务内新增。 |
| 来源依据 | 建议 | 生成该节点时引用的业务输入、文档样本或管理员说明。 |
| 节点状态 | 必填 | 草稿、待生效、已生效、已修改、已停用、已删除。 |
| 替代节点 ID | 建议 | 停用或删除后建议迁移到哪个节点。 |
| 停用或删除原因 | 建议 | 影响分析和审计需要。 |
| 创建人 | 建议 | 审计字段。 |
| 更新人 | 建议 | 审计字段。 |
| 创建时间 | 必填 | 审计字段。 |
| 更新时间 | 必填 | 审计字段。 |

#### 3.8.4 文档标签结果字段

标签结果表示某篇文档与某个三级叶子标签之间的绑定关系。

| 字段 | 必要性 | 说明 |

## 4. 已确认业务约束

### 4.1 模块边界约束

1. 本模块是现有项目中的业务功能模块，不是完整项目或独立产品。
2. 本模块不定义知识库创建、用户体系、权限体系、文件上传、全文检索等通用能力。
3. 本模块只处理文档入库后与 Summary、Taxonomy、自动打标、标签维护、版本治理和检索消费相关的业务流程。
4. 本模块输出的 Summary、Taxonomy 和标签结果需要被当前项目的知识库能力消费。

### 4.2 Taxonomy 归属约束

1. Taxonomy 应按企业或租户维度生效。
2. 同一企业内默认只有一套当前生效 Taxonomy。
3. 不同企业可以拥有不同的 Taxonomy。
4. 系统默认模板是跨企业共享的候选模板，不等同于企业当前生效 Taxonomy。
5. 企业可从默认模板、行业模板和企业自定义节点中组合形成自己的生效 Taxonomy。

### 4.3 Taxonomy 结构约束

1. MVP 阶段 Taxonomy 固定为三级结构。
2. 产品交互和业务校验不支持两级或四级及以上的正式生效 Taxonomy。
3. 文档最终打标结果只允许绑定三级叶子标签。
4. 一级标签和二级标签只作为分类层级、聚合展示、批量选择和影响分析使用，不作为文档最终绑定标签。
5. 新增标签必须以完整三级路径表达。
6. 每个节点必须包含节点说明，节点说明用于界定语义、适用范围和与相近节点的区别。
7. 如果新增内容涉及一级或二级层级变化，也必须同时形成可用于文档绑定的三级叶子标签。
8. Taxonomy 生成或修改后需要人工确认才能正式生效。
9. 批量打标必须优先基于当前任务绑定的生效 Taxonomy 版本执行。

### 4.4 默认模板启用约束

1. 系统应提供企业通用默认 Taxonomy 模板。
2. 管理员可以按三级叶子节点粒度选择保留或不选择默认模板中的分类。
3. 管理员选择一级或二级节点时，系统应明确展示其下包含的三级叶子节点，并允许继续细化选择。
4. 只有至少包含一个已选择三级叶子节点的一级或二级节点，才可进入企业 Taxonomy 草稿。
5. 未被选择的默认模板节点不进入企业生效 Taxonomy。
6. 默认模板节点进入企业 Taxonomy 后，可被管理员修改名称、说明、示例和适用范围。
7. 默认模板节点被企业采用后，后续企业内变更不应反向影响平台默认模板。

### 4.5 LLM 生成 Taxonomy 约束

1. 管理员可以输入行业、业务范围、组织结构、知识库类型、分类偏好等信息，让系统生成企业特色 Taxonomy 草稿。
2. 系统也可以基于企业已有文档标题、目录、Summary、元数据和内容样本生成候选 Taxonomy。
3. LLM 生成结果必须为三级结构。
4. LLM 生成结果只作为草稿，不得自动成为生效 Taxonomy。
5. LLM 生成结果需要经过结构校验和健康自检，识别重复、层级错位、粒度不均、定义缺失等问题。
6. 管理员可以删除、修改、补充或组合选择生成结果中的候选节点，但系统不提供自动标签合并能力。
7. 管理员确认后，生成结果才可进入正式版本。

### 4.6 Taxonomy 版本约束

1. 初始 Taxonomy 确认生效后，形成第一个 Taxonomy 版本。
2. 每次 Taxonomy 发生正式变更，均需要生成新的 Taxonomy 版本。
3. Taxonomy 版本至少应记录版本号、变更原因、变更来源、变更时间、变更内容和生效状态。
4. 批量打标任务启动时必须绑定当时的 Taxonomy 版本。
5. 批量打标任务执行过程中，不应被外部新版本 Taxonomy 中途影响。
6. 文档标签结果需要记录其使用的 Taxonomy 版本。
7. 当文档标签结果所依赖的三级叶子标签被删除、停用或发生语义重定义时，文档应进入未打标签池或待重新打标池。
8. 并发批量打标任务产生的新增节点，在各任务完成后应基于最新已生效 Taxonomy 进行整理、去重和落版。

### 4.7 Summary 约束

1. Summary 为文档入库后的自动处理能力，不要求用户手动单独发起。
2. Summary 结果需要可查看，并可作为后续批量打标的数据来源之一。
3. Summary 结果允许人工编辑，编辑后的人工版本优先级高于 AI 自动生成版本。
4. Summary 被人工修改后，该文档原有标签结果应失效，文档进入未打标签池或待重新打标池。

### 4.8 批量打标约束

1. 批量打标支持基于原文内容执行，也支持基于 Summary 执行。
2. 单篇文档可命中多个三级叶子标签。
3. 未开启 Taxonomy 优化时，系统只基于当前任务绑定的 Taxonomy 版本输出推荐叶子标签或未匹配结果。
4. 开启 Taxonomy 优化时，系统才允许输出候选新增三级叶子标签。
5. 打标结果应优先复用当前 Taxonomy 中已有的三级叶子标签。
6. 新增叶子标签是例外处理，不是默认行为。
7. “无法覆盖”“冗余”“优化强度”等判断口径由 Prompt 和模型处理策略承载，本文档不展开具体 Prompt 规则。
8. Prompt 需要在开发和测试阶段通过业务样例人工打磨确认，验收时按确认后的 Prompt 样例验证。

### 4.9 多 Agent 协作约束

1. 为降低单个 LLM 处理节点的任务复杂度，本模块采用多 Agent 协作的业务分工。
2. 打标 Agent 负责基于当前 Taxonomy 版本对文档进行标签推荐，并识别当前体系无法覆盖的内容。
3. Taxonomy 健康自检 Agent 负责判断候选新增节点是否冗余、是否符合三级结构、是否适合加入 Taxonomy。
4. Taxonomy 变更影响分析 Agent 或系统规则负责识别节点修改、删除、停用对已打标文档的影响。
5. 打标 Agent 不负责最终决定 Taxonomy 是否新增、修改或删除。
6. 健康自检 Agent 不删除已有节点，不合并已有节点。
7. 健康自检 Agent 不决定 Taxonomy 版本是否更新，版本更新由系统在任务完成或人工维护完成后处理。
8. 启用 Taxonomy 优化前的强提醒确认，视为管理员授权系统在自检通过后将新节点自动应用到触发该节点的文档，并在任务完成后统一生成新版本。

### 4.10 Taxonomy 维护约束

1. 管理员可以人工新增、修改、移动、停用或删除 Taxonomy 节点。
2. 管理员人工新增三级叶子标签时，也必须经过健康自检 Agent 判断是否冗余。
3. 若人工新增标签与已有标签语义重复或高度相似，系统应提示复用已有标签，而不是直接新增。
4. 本模块 MVP 不提供自动标签合并能力。
5. 已有标签即使语义接近，也不自动合并或删除；可通过修改标签说明、层级定义等方式澄清其适用边界。
6. 删除任一级别节点都属于高影响级联删除操作。
7. 删除一级节点时，其下所有二级节点、三级叶子标签一并删除或失效。
8. 删除二级节点时，其下所有三级叶子标签一并删除或失效。
9. 删除三级叶子标签时，仅删除或失效该三级叶子标签本身。
10. 删除节点前，系统必须检查该节点及其下级叶子标签是否已经应用在文档上。
11. 若被删除范围内的三级叶子标签已经应用在文档上，删除前必须强提示管理员：受影响文档将失去有效三级标签，并进入未打标签池或待重新打标池。
12. Taxonomy 的新增、修改、移动、停用、删除行为均需要保留变更记录，并生成新的 Taxonomy 版本。

### 4.11 节点变更影响约束

不同类型的节点变更对已打标文档影响不同：

1. 仅修改节点展示名称，且不改变节点语义时，已打标文档不需要重新分类，但需要展示最新名称和路径。
2. 修改节点定义、适用范围、正反例、关键词或同义词时，系统应提示可能影响分类边界，并将相关文档标记为待复核或待重新打标。
3. 移动节点挂载位置时，系统应更新完整路径，并提示该节点及其下属文档可能需要重新分类。
4. 停用三级叶子标签时，历史标签结果可保留但不再用于新打标；受影响文档应进入待复核或待重新打标池。
5. 删除三级叶子标签时，绑定该标签的文档应失去该标签并进入未打标签池或待重新打标池。
6. 删除一级或二级节点时，其下所有已绑定文档均应进入未打标签池或待重新打标池。
7. 管理员执行高影响变更前，系统必须展示影响分析结果和二次确认。

### 4.12 未打标签池约束

1. 文档因 Summary 修改、标签删除、标签停用、标签语义重定义、打标未匹配或低置信度进入未打标签池或待重新打标池。
2. 进入池子的文档可由系统自动加入待打标队列。
3. 管理员也可以从池子中选择文档并批量触发重新打标。
4. 重新打标时应拉取最新已生效 Taxonomy 版本。

### 4.13 检索消费约束

1. 生效的三级叶子标签应能被当前知识库检索、筛选和展示消费。
2. 检索时可以通过用户选择的一级、二级或三级节点缩小范围，但最终应转换为三级叶子标签过滤。
3. 系统可以尝试从用户 query 中自动识别相关 Taxonomy 节点，用于加速和缩小检索范围。
4. Query 自动识别应采用层级递进逻辑：优先识别三级叶子标签；若三级候选置信度不足，则扩大到二级节点；若二级候选置信度仍不足，则扩大到一级节点。
5. Query 自动命中 Taxonomy 节点不得成为检索的唯一入口。
6. 自动命中的最低置信度阈值应支持管理员配置，例如默认 0.8。
7. 高置信度命中时可以使用硬过滤。
8. 中置信度命中时应使用软过滤或自动放宽检索。
9. 低置信度或无命中时应回退到原有知识库检索。
10. Taxonomy 检索过滤不得绕过原有文档权限控制。

## 5. 核心业务流程

### 5.1 Taxonomy 初始化流程

1. 系统提供企业通用默认 Taxonomy 模板。
2. 管理员查看默认模板，并按三级叶子节点选择保留或不选择。
3. 管理员可输入行业、企业业务、组织结构、知识库内容范围等信息。
4. 系统基于输入生成行业或企业特色 Taxonomy 草稿。
5. 系统对草稿执行结构校验和健康自检。
6. 管理员预览默认模板选择结果和企业生成草稿。
7. 管理员修改节点名称、定义、正反例、关键词、挂载位置等信息。
8. 管理员确认后，形成企业初始 Taxonomy 版本。
9. 未确认的 Taxonomy 草稿不得用于正式批量打标。

### 5.2 文档处理与打标流程

1. 用户批量导入文本文件。
2. 系统对导入文件进行解析，并自动生成 Summary。
3. Summary 完成后，文档进入可打标状态。
4. 管理员选择目标文档发起批量打标任务。
5. 系统为批量打标任务绑定当前生效 Taxonomy 版本。
6. 管理员选择打标来源为原文或 Summary。
7. 管理员选择是否在本次打标过程中启用 Taxonomy 优化。
8. 打标 Agent 基于任务绑定的 Taxonomy 版本输出文档三级叶子标签。
9. 若当前 Taxonomy 无法覆盖文档内容，且已开启优化，打标 Agent 输出候选新增三级叶子标签及完整三级路径。
10. 健康自检 Agent 对候选新增叶子标签进行冗余、结构和挂载位置判断。
11. 若候选新增标签与已有标签冗余，自检 Agent 输出建议复用的已有三级叶子标签。
12. 若候选新增标签不冗余，系统先将该标签作为本任务新增标签结果，并将该新标签应用到触发该标签的文档。
13. 任务完成后，系统汇总本任务内全部新增标签结果，整理去重后生成新的 Taxonomy 版本。
14. 系统输出打标结果、版本变化、未匹配内容和受影响文档。

### 5.3 Taxonomy 维护与重分类流程

1. 管理员查看当前生效 Taxonomy。
2. 管理员发起新增、修改、移动、停用或删除节点操作。
3. 新增节点进入健康自检流程。
4. 修改、移动、停用或删除节点进入影响分析流程。
5. 系统展示影响范围，包括受影响节点、受影响三级叶子标签、受影响文档数量和代表性文档。
6. 高影响操作需要管理员二次确认。
7. 操作确认后生成新的 Taxonomy 版本。
8. 受影响文档进入未打标签池、待复核池或待重新打标池。
9. 管理员可批量触发重新打标。
10. 重新打标使用最新已生效 Taxonomy 版本。

### 5.4 检索使用流程

1. 普通用户在权限范围内检索知识库。
2. 用户可手动选择一级、二级或三级 Taxonomy 节点作为筛选条件。
3. 系统将一级或二级筛选转换为其下已生效三级叶子标签集合。
4. 系统也可对 query 进行轻量级 Taxonomy 节点识别。
5. 系统优先尝试命中三级叶子标签；若三级叶子标签置信度不足，则递进回退到对应二级节点；若二级节点置信度仍不足，则继续回退到一级节点。
6. 当某一层级节点达到管理员配置的置信度阈值时，系统可将该节点转换为其下已生效三级叶子标签集合，用于缩小检索范围。
7. 中低置信度、检索结果不足或 Taxonomy 质量不足时，系统应自动放宽或回退到普通检索。
8. 检索结果应展示文档所属 Taxonomy 路径。

### 5.5 关键原则

1. Summary、Taxonomy 版本、打标结果、变更影响和检索消费需要形成完整链路。
2. 文档最终只绑定三级叶子标签。
3. 标签推荐优先复用当前已生效 Taxonomy，新增标签是例外处理。
4. Taxonomy 优化必须经过“候选新增标签 -> 健康自检 -> 任务内应用或复用 -> 任务完成后统一落版”的流程。
5. 系统需要区分“任务绑定版本”“最新生效版本”和“文档标签结果版本”。
6. 涉及 Taxonomy 结构或语义变化的操作，必须保留变更记录和影响提示。
7. Query 自动命中 Taxonomy 只用于增强检索，不应降低原有检索召回。

## 6. 功能一：批量文本文件的 AI 自动 Summary

### 6.1 功能目标

在文本文件批量入库后，系统自动为每篇文档生成 Summary，供用户查看，并支持后续批量打标直接使用 Summary 作为输入内容。

### 6.2 功能定义

1. 支持对批量导入的文本文件逐篇生成 Summary。
2. Summary 生成应属于文档入库处理链路中的标准步骤。
3. 每篇文档应保留对应的 Summary 结果。
4. Summary 结果需要支持列表查看。
5. Summary 结果需要允许人工编辑，编辑后以人工版本为准。
6. Summary 修改后，需要触发文档回归未打标签池或待重新打标池。

### 6.3 输入

1. 已导入平台的文本文件。
2. 文档基础信息，包括文档标题、所属知识库、所属目录等。

### 6.4 输出

每篇文档至少需要产出以下结果：

1. 文档标题。
2. Summary 内容。
3. Summary 生成时间。
4. Summary 状态。
5. 是否为人工编辑版本。
6. 当前标签状态。

### 6.5 业务规则

1. Summary 自动执行，无需用户单独创建 Summary 任务。
2. Summary 完成后，文档才进入可用于打标的状态。
3. Summary 查看场景需支持按知识库、目录筛选。
4. 若 Summary 失败，需要保留失败状态和失败标识。
5. 人工编辑 Summary 后，后续若选择基于 Summary 打标，应使用最新保存版本。
6. Summary 被人工修改后，文档当前标签结果失效，文档进入未打标签或待重新打标状态。

### 6.6 用户可感知结果

1. 用户可以看到每篇文档是否已生成 Summary。
2. 用户可以查看 Summary 内容。
3. 用户可以修改 Summary 内容。
4. 用户可以看到 Summary 修改后文档已回归未打标签池或待重新打标池。
5. 用户在批量打标时可以明确选择是否基于 Summary 执行。

---

## 7. 功能二：默认模板选择与企业 Taxonomy 生成

### 7.1 功能目标

帮助知识库管理员快速形成企业可用的三级 Taxonomy。系统先提供通用默认模板，再允许管理员结合行业和企业知识库内容生成企业特色分类，最终由管理员确认形成生效版本。

### 7.2 功能定义

1. 支持系统提供企业通用默认 Taxonomy 模板。
2. 支持管理员按三级叶子节点选择保留或不选择默认模板分类。
3. 支持管理员输入行业、业务范围、组织结构、分类偏好等业务提示词。
4. 支持系统基于业务提示词和知识库内容样本生成企业特色 Taxonomy 草稿。
5. 生成结果必须为三级结构。
6. Taxonomy 应包含节点名称、层级关系、节点定义、适用范围、正例、反例、关键词和同义词。
7. 节点说明为必填内容。
8. Taxonomy 草稿生成后需支持预览、修改和删除。
9. 管理员确认后正式生效，并形成初始 Taxonomy 版本。

### 7.3 输入

1. 系统默认 Taxonomy 模板。
2. 管理员选择保留的默认模板三级节点。
3. 管理员填写的业务提示词。
4. 行业语境、企业业务范围、组织结构、知识库范围、分类偏好等业务语义信息。
5. 可选的文档标题、目录、Summary、元数据或内容样本。

### 7.4 输出

Taxonomy 草稿至少应包含：

1. 一级节点。
2. 二级节点。
3. 三级叶子节点。
4. 每个节点的名称。
5. 每个节点的说明或定义。
6. 每个节点的适用范围。
7. 三级叶子节点的正例和反例。
8. 三级叶子节点的关键词和同义词。
9. 节点来源。
10. Taxonomy 版本号或草稿标识。

### 7.5 业务规则

1. Taxonomy 必须固定为三级。
2. 默认模板只是候选，不自动成为企业生效 Taxonomy。
3. 管理员必须明确选择保留哪些默认模板三级节点。
4. 企业特色生成结果在确认前属于草稿状态。
5. 未确认的 Taxonomy 草稿不能用于正式批量打标。
6. Taxonomy 一旦确认，即作为当前企业的生效 Taxonomy 使用。
7. Taxonomy 生成结果需支持树形查看。
8. Taxonomy 生成结果应尽量避免同级重复、语义重叠、层级错位和粒度不均。
9. 标签说明缺失时，不允许确认 Taxonomy 生效。
10. 初始 Taxonomy 确认后，后续打标与维护均应基于版本机制进行。

### 7.6 用户可感知结果

1. 管理员可以看到系统默认 Taxonomy 模板。
2. 管理员可以按三级叶子节点选择保留或删除默认分类。
3. 管理员可以输入业务提示词发起企业特色 Taxonomy 生成。
4. 管理员可以看到树形 Taxonomy 草稿。
5. 管理员可以查看每个节点的定义、正例、反例和关键词。
6. 管理员可以确认、取消或继续编辑草稿。
7. 管理员可以清楚区分“默认模板”“生成草稿”和“已生效 Taxonomy”。
8. 管理员可以看到当前生效 Taxonomy 版本。

## 8. 功能三：批量 AI 打标

### 8.1 功能目标

针对已完成解析和 Summary 处理的文档，基于当前任务绑定的 Taxonomy 版本批量生成三级叶子标签结果。

### 8.2 功能定义

1. 支持选择多篇文档发起批量打标。
2. 支持按知识库、目录、关键词、当前标签状态等方式筛选目标文档。
3. 支持选择打标来源为原文或 Summary。
4. 支持基于任务绑定的 Taxonomy 版本生成标签结果。
5. 支持单篇文档返回多个三级叶子标签。
6. 支持查看每个推荐标签的置信度。
7. 支持识别“当前 Taxonomy 无法覆盖”的文档或内容。
8. 支持在开启 Taxonomy 优化时输出候选新增三级叶子标签。
9. 支持将自检通过的新叶子标签自动应用到触发该标签的文档。
10. 支持在任务完成后统一生成新的 Taxonomy 版本。

### 8.3 输入

1. 待打标文档集合。
2. 打标来源选择：原文或 Summary。
3. 任务绑定的 Taxonomy 版本。
4. 是否开启 Taxonomy 优化。
5. 优化强度配置。

### 8.4 输出

每篇文档至少应返回以下信息：

1. 文档标题。
2. 三级叶子标签列表。
3. 每个叶子标签的完整三级路径。
4. 每个标签的置信度。
5. 标签来源。
6. 未匹配内容或未覆盖原因。
7. 候选新增标签，仅在开启 Taxonomy 优化时输出。
8. 标签结果使用的 Taxonomy 版本。
9. 若有任务内新增标签，需标记其待落版状态。
10. 当前标签状态。

### 8.5 业务规则

1. 仅已完成解析和 Summary 处理的文档可进入打标流程。
2. 若选择基于 Summary 打标，则应使用当前可用的 Summary 内容。
3. 打标 Agent 必须优先从任务绑定的 Taxonomy 版本中选择三级叶子标签。
4. 普通打标结果只能绑定已存在的三级叶子标签。
5. 未开启 Taxonomy 优化时，打标 Agent 不输出新增标签，只输出低置信度、未匹配或需处理状态。
6. 开启 Taxonomy 优化时，打标 Agent 可以输出候选新增三级叶子标签，但候选新增标签必须进入健康自检流程。
7. 候选新增标签自检冗余时，系统复用已有三级叶子标签。
8. 候选新增标签自检不冗余时，系统将该标签作为本任务新增标签结果，并将该标签应用到触发该标签的文档。
9. 本任务内新增标签只应用于触发该标签的文档，不参与本任务后续其他文档的打标候选。
10. 任务完成后，系统统一整理本任务内新增标签结果，并生成新的 Taxonomy 版本。
11. 后续新任务重新打标时，应拉取数据库中最新已生效 Taxonomy 版本。

### 8.6 用户可感知结果

1. 管理员可以选择文档并发起任务。
2. 管理员可以看到任务执行进度。
3. 管理员可以逐篇查看三级叶子标签结果。
4. 管理员可以看到哪些内容没有被当前 Taxonomy 良好覆盖。
5. 管理员可以看到本次任务使用的 Taxonomy 版本。
6. 管理员可以明确区分“已有叶子标签”“候选新增标签”“任务内新增标签”和“已落版标签”。

## 9. 功能四：批量打标中的 Taxonomy 优化

### 9.1 功能目标

在批量打标过程中，对于当前 Taxonomy 无法良好覆盖的文档内容，支持输出候选新增三级叶子标签，并通过健康自检 Agent 判断是否冗余。若自检通过，系统先将新标签应用到触发文档，并在任务完成后统一整理生成新的 Taxonomy 版本。

### 9.2 功能定义

1. 批量打标时支持开启或关闭 Taxonomy 优化。
2. 开启后支持设置优化强度。
3. 优化强度的具体判定口径由 Prompt 和模型处理策略承载。
4. 开启后在正式执行前必须进行二次确认。
5. 二次确认需要明确说明：本次任务可能产生新增叶子标签、将新标签自动应用到触发文档，并在任务完成后统一生成新 Taxonomy 版本。
6. 系统需要识别未能有效归入现有 Taxonomy 的文档或内容。
7. 打标 Agent 需要输出候选新增三级叶子标签、完整三级路径和来源依据。
8. 健康自检 Agent 需要判断候选新增标签是否与已有标签冗余。
9. 若候选新增标签冗余，自检 Agent 应输出建议复用的已有三级叶子标签。
10. 若候选新增标签不冗余，系统应将其纳入本任务新增标签结果。
11. 任务完成后，系统应统一整理本任务新增标签结果，并生成新的 Taxonomy 版本。

### 9.3 多 Agent 职责边界

打标 Agent 负责：

1. 基于当前 Taxonomy 版本对文档进行打标。
2. 判断文档内容是否能被当前 Taxonomy 覆盖。
3. 在开启优化时提出候选新增三级叶子标签。
4. 给出候选新增三级叶子标签的完整路径、来源依据和适用文档。

健康自检 Agent 负责：

1. 判断候选新增标签是否与已有标签语义重复。
2. 判断候选新增标签是否只是已有标签的同义表达、细粒度重复或命名变体。
3. 判断候选新增标签是否符合三级 Taxonomy 结构。
4. 判断候选新增标签的建议挂载位置是否合理。
5. 输出“复用已有标签”“新增标签”“拒绝新增”或“需处理”的判断。
6. 不负责合并已有标签。
7. 不负责删除已有标签。
8. 不负责决定 Taxonomy 版本更新时间。

管理员负责：

1. 确认是否开启 Taxonomy 优化。
2. 维护 Taxonomy。
3. 执行删除、停用、重定义等高影响操作。

### 9.4 输入

1. 批量打标任务。
2. 任务绑定的 Taxonomy 版本。
3. 是否开启 Taxonomy 优化。
4. 优化强度配置。
5. 打标 Agent 输出的候选新增三级叶子标签。
6. 候选新增标签对应的文档和来源依据。

### 9.5 输出

当存在候选新增标签时，至少应输出：

1. 候选新标签名称。
2. 完整三级路径。
3. 建议挂载位置。
4. 建议来源依据。
5. 建议置信度。
6. 自检判断结果。
7. 若冗余，建议复用的已有三级叶子标签。
8. 若不冗余，任务内新增标签结果。
9. 关联文档数量和文档列表。
10. 标签应用结果。
11. 任务完成后的 Taxonomy 版本。

### 9.6 业务规则

1. Taxonomy 优化为可选项，不开启时仅执行普通批量打标。
2. 开启 Taxonomy 优化时，开始执行前必须弹出强提醒确认。
3. 强提醒确认后，系统可在自检通过时将新增标签自动应用到触发文档。
4. 候选新增标签必须先经过健康自检 Agent 判断。
5. 自检判断为冗余时，不得直接新增标签，应复用已有三级叶子标签。
6. 自检判断为不冗余时，系统先记录为任务内新增标签，不立即生成新版本。
7. 任务内新增标签应自动用于触发该标签的原文档。
8. 任务内新增标签不参与本任务后续其他文档的打标候选。
9. 任务完成后，系统对全部任务内新增标签进行整理去重，并统一生成新的 Taxonomy 版本。
10. 并发任务完成时，应基于最新已生效 Taxonomy 重新整理本任务新增标签，再生成新版本。
11. 自检判断为拒绝新增或需处理时，文档进入未匹配或未打标签状态。
12. 启用优化时，应保留未匹配文档或未匹配内容的收集结果。
13. Taxonomy 更新行为需要保留变更记录。

### 9.7 用户可感知结果

1. 管理员可以决定本次打标是否启用 Taxonomy 优化。
2. 管理员可以在开始前收到明确确认提醒。
3. 管理员可以看到未匹配内容和候选新增标签。
4. 管理员可以看到健康自检 Agent 对候选新增标签的冗余判断。
5. 管理员可以看到系统复用了哪个已有三级叶子标签。
6. 管理员可以看到任务完成后生成的 Taxonomy 版本。
7. 管理员可以看到新标签自动应用到了哪些文档。

## 10. 功能五：Taxonomy 健康自检与人工维护

### 10.1 功能目标

支持管理员在 Taxonomy 生效后进行人工维护，同时通过健康自检 Agent 降低新增冗余标签的风险，并通过影响分析和重分类机制保证企业知识分类持续准确。

### 10.2 功能定义

1. 支持管理员查看当前已生效 Taxonomy。
2. 支持管理员人工新增三级叶子标签。
3. 支持管理员修改节点名称、说明、正例、反例、关键词、同义词或挂载关系。
4. 支持管理员停用节点。
5. 支持管理员删除节点。
6. 支持管理员查看节点变更影响分析。
7. 人工新增标签必须经过健康自检 Agent 判断。
8. 删除、停用、重定义节点前必须检查该节点及其下级叶子标签是否已被文档使用。
9. 高影响操作必须展示影响提示和二次确认。
10. Taxonomy 维护结果需保留变更记录，并生成新的 Taxonomy 版本。
11. 本模块 MVP 不支持自动标签合并。
12. 节点说明为新增和修改节点时的必填信息。

### 10.3 人工新增标签规则

1. 管理员新增标签时，需要填写标签名称、完整三级路径和标签说明。
2. 新增标签提交后，先进入自检流程。
3. 自检 Agent 需要判断该标签是否与已有标签重复、相似或可被已有标签覆盖。
4. 若自检结果为冗余，系统应提示复用的已有三级叶子标签，并说明原因。
5. 若自检结果为不冗余，系统允许新增该三级叶子标签，并生成新的 Taxonomy 版本。
6. 人工新增标签不得破坏三级 Taxonomy 结构。
7. 人工新增标签只更新 Taxonomy，不自动影响已有文档，除非管理员主动发起重新打标。

### 10.4 人工修改节点规则

1. 管理员可以修改节点名称和节点说明。
2. 修改节点名称或说明时，应保留原节点的历史记录。
3. 若修改会造成与其他节点语义重叠，系统应提示潜在冗余风险。
4. 若修改挂载关系，需要保证 Taxonomy 仍为三级结构。
5. 修改后的 Taxonomy 需要生成新的版本。
6. 已有标签不因语义接近而自动合并。
7. 修改后节点说明不得为空。
8. 修改节点语义边界时，系统应提示该节点及相关文档可能需要重新打标。

### 10.5 人工停用节点规则

1. 停用节点表示该节点不再参与后续新文档打标。
2. 停用一级节点时，其下所有二级节点和三级叶子标签一并停用。
3. 停用二级节点时，其下所有三级叶子标签一并停用。
4. 停用三级叶子标签时，仅停用该叶子标签本身。
5. 已停用节点的历史标签结果可保留展示，但应标记为依赖已停用标签。
6. 停用已使用节点前必须展示影响文档数量和代表性文档。
7. 停用后受影响文档应进入待复核或待重新打标池。
8. 停用操作需要生成新的 Taxonomy 版本。

### 10.6 人工删除节点规则

1. 删除任一级别节点都属于高影响操作。
2. 管理员删除节点前，系统必须检查该节点及其下级叶子标签是否已应用在文档上。
3. 删除一级节点时，其下所有二级节点、三级叶子标签一并删除或失效。
4. 删除二级节点时，其下所有三级叶子标签一并删除或失效。
5. 删除三级叶子标签时，仅删除或失效该三级叶子标签本身。
6. 任一级别节点删除后，该节点所在的有效标签路径不再存在。
7. 若被删除范围内的三级叶子标签未应用在任何文档上，可进入普通删除确认。
8. 若被删除范围内的三级叶子标签已应用在文档上，系统必须展示强提示。
9. 强提示至少应包含：删除范围、受影响三级叶子标签数量、受影响文档数量、代表性文档列表、删除后的影响说明。
10. 删除后的影响说明必须明确：受影响文档将失去有效三级标签，并回归未打标签池或待重新打标池。
11. 删除已应用标签必须由管理员二次确认，不允许静默删除。
12. 删除后，受影响文档应进入未打标签或待重新打标状态。
13. 删除后需要生成新的 Taxonomy 版本。

### 10.7 标签合并规则

1. 本模块 MVP 不提供自动标签合并能力。
2. 已有标签即使存在语义接近、同词多义或层级释义差异，也应保留。
3. 对于容易误解的已有标签，优先通过修改标签说明、正反例或层级定义来澄清，而不是合并或删除。
4. 自检 Agent 判断新增标签冗余时，只建议复用已有标签，不执行合并。

### 10.8 输出

Taxonomy 维护至少应输出：

1. 变更类型：新增、修改、移动、停用、删除。
2. 变更前内容。
3. 变更后内容。
4. 自检结果。
5. 冗余判断依据。
6. 影响文档数量。
7. 受影响三级叶子标签列表。
8. 受影响文档列表。
9. 建议处理方式：无需处理、待复核、待重新打标、进入未打标签池。
10. Taxonomy 版本变化。
11. 变更生效时间。
12. 节点说明。

### 10.9 用户可感知结果

1. 管理员可以维护当前 Taxonomy。
2. 管理员新增标签时可以看到是否存在冗余。
3. 管理员可以看到系统建议复用的已有三级叶子标签。
4. 管理员删除或停用节点时可以看到级联影响范围。
5. 管理员可以看到该操作影响哪些文档。
6. 管理员可以明确知道删除、停用或重定义已使用标签会导致相关文档回归未打标签池、待复核池或待重新打标池。
7. 管理员可以查看 Taxonomy 变更记录和版本变化。

---

## 11. 功能六：展示打标结果、Taxonomy 版本与变更影响

### 11.1 功能目标

让管理员能够清晰查看批量打标任务的执行进度、每篇文档的三级叶子标签结果、候选新增标签、自检判断、Taxonomy 版本变化、节点变更影响以及待重新打标文档。

### 11.2 功能定义

1. 支持展示批量打标任务的整体状态。
2. 支持展示任务处理进度。
3. 支持展示每篇文档的三级叶子标签结果。
4. 支持展示成功、失败、未匹配、未打标签、待复核、待重新打标等状态。
5. 支持展示本次任务绑定的 Taxonomy 版本。
6. 支持在启用 Taxonomy 优化时同步展示候选新增标签、自检结果和任务完成后的版本。
7. 支持展示 Taxonomy 变更对文档的影响。
8. 支持展示未打标签池和待重新打标池及其重新打标入口。
9. 支持展示文档在检索结果中的完整 Taxonomy 路径。

### 11.3 展示维度

任务维度至少包括：

1. 任务状态。
2. 文档总数。
3. 已处理数量。
4. 成功数量。
5. 失败数量。
6. 未匹配数量。
7. 当前处理进度。
8. 任务绑定 Taxonomy 版本。
9. 任务完成后 Taxonomy 版本。
10. 任务内新增标签数量。

文档维度至少包括：

1. 文档标题。
2. 三级叶子标签。
3. 标签完整三级路径。
4. 置信度。
5. 标签来源。
6. 未匹配原因。
7. Taxonomy 版本。
8. 当前标签状态。
9. 是否来自任务内新增标签。

Taxonomy 优化维度至少包括：

1. 候选新增标签。
2. 完整三级路径。
3. 来源文档。
4. 来源依据。
5. 自检判断。
6. 复用的已有三级叶子标签。
7. 任务内新增标签结果。
8. 任务完成后的 Taxonomy 版本。
9. 自动应用文档列表。

Taxonomy 变更影响维度至少包括：

1. 变更节点。
2. 变更类型。
3. 级联影响范围。
4. 影响三级叶子标签数量。
5. 影响文档数量。
6. 影响文档列表。
7. 回归未打标签池、待复核池或待重新打标池的文档列表。

未打标签池和待重新打标池维度至少包括：

1. 文档标题。
2. 进入池子的原因。
3. 原 Taxonomy 版本。
4. 当前最新 Taxonomy 版本。
5. 原绑定标签。
6. 是否可自动加入待打标队列。
7. 是否可被管理员批量选择重新打标。

### 11.4 业务规则

1. 打标结果展示应区分任务层、文档层、Taxonomy 优化层和变更影响层。
2. 展示结果应区分任务绑定版本、最新生效版本和文档标签结果版本。
3. 若文档处理失败，应可识别失败状态。
4. 若任务尚未完成，应持续展示处理中状态和进度。
5. 若文档因 Summary 修改、标签删除、标签停用或标签重定义导致原标签失效，应显示未打标签、待复核或待重新打标状态。
6. 若启用 Taxonomy 优化，应展示自动新增标签及自动应用结果。
7. 未打标签池和待重新打标池中的文档重新打标时，应使用最新已生效 Taxonomy 版本。

### 11.5 用户可感知结果

1. 管理员可以知道任务是否已开始、进行中、已完成或已取消。
2. 管理员可以看到每篇文档具体绑定了哪些三级叶子标签。
3. 管理员可以看到本次任务使用了哪个 Taxonomy 版本。
4. 管理员可以看到本次任务是否伴随 Taxonomy 优化及其结果。
5. 管理员可以看到健康自检 Agent 对新增标签的判断。
6. 管理员可以看到 Taxonomy 变更影响了哪些文档。
7. 管理员可以看到哪些文档已回归未打标签池、待复核池或待重新打标池。
8. 管理员可以从池子中批量触发重新打标。

## 12. 功能七：Taxonomy 在检索与筛选中的使用

### 12.1 功能目标

让 Taxonomy 不只服务于后台治理，也能在知识库使用过程中提升筛选、导航和检索范围收敛能力。

Query 自动命中 Taxonomy 节点、命中结果转换为 OpenSearch 标签过滤、检索 fallback、可配置项和分阶段交付细则，见 `docs/new/taxonomy-search.md`。本章节只保留核心业务边界。

### 12.2 功能定义

1. 支持用户按 Taxonomy 一级、二级、三级节点筛选文档。
2. 支持检索结果展示文档所属完整 Taxonomy 路径。
3. 支持系统从 query 中自动识别可能相关的 Taxonomy 节点。
4. 支持根据识别层级、识别置信度和管理员配置决定是否使用 Taxonomy 过滤。
5. 支持在 Taxonomy 过滤结果不足时自动放宽检索范围。
6. 支持管理员查看哪些 Taxonomy 节点被高频用于筛选或检索。

### 12.3 业务规则

1. 用户手动选择一级节点时，系统应转换为其下所有已生效三级叶子标签。
2. 用户手动选择二级节点时，系统应转换为其下所有已生效三级叶子标签。
3. 用户手动选择三级节点时，系统直接使用该叶子标签。
4. Query 自动命中 Taxonomy 时，应支持展示建议、软过滤、硬过滤和回退普通检索。
5. Query 自动命中应支持从三级叶子标签递进扩大到二级、一级节点，但具体阈值和回退逻辑由 `docs/new/taxonomy-search.md` 定义。
6. 自动命中的置信度阈值应可配置，默认建议为 0.8。
7. 高置信度命中可作为硬过滤。
8. 中置信度命中可作为软过滤，并在结果不足时自动放宽。
9. 低置信度命中不得强制过滤。
10. Taxonomy 过滤必须与原有权限过滤共同生效，不得扩大用户可见范围。
11. Taxonomy 过滤失败或超时时，不应阻断原有检索链路。

### 12.4 用户可感知结果

1. 用户可以通过 Taxonomy 树快速缩小搜索范围。
2. 用户可以看到检索结果所属分类路径。
3. 用户在 query 命中分类时，可以看到系统建议的分类范围。
4. 当系统自动放宽检索范围时，用户应能理解结果并非严格限制在某个分类下。

3. 解析成功。
4. 解析失败。
5. 待 Summary。

10. 打标中。
11. 打标成功。
12. 未匹配。
13. 低置信度待确认。
14. 待复核。
15. 待重新打标。
16. 已打标签。

### 13.2 任务状态

批量打标任务至少应覆盖以下状态：

1. 待执行。
2. 执行中。
3. 已完成。
4. 已取消。
5. 执行失败。

1. 待自检。
2. 自检中。
3. 已复用已有标签。
4. 任务内新增。
5. 待统一落版。
6. 已落版。
7. 已拒绝新增。
8. 需处理。

### 13.4 Taxonomy 节点状态

Taxonomy 节点至少应覆盖以下状态：

1. 草稿。

4. 已修改。
5. 已停用。
6. 已删除。

Taxonomy 版本至少应覆盖以下状态：

1. 草稿。
2. 待生效。
3. 已生效。
4. 已被新版本替代。

Taxonomy 变更记录至少应覆盖以下状态：

1. 待执行。

4. 已取消。

6. 复核中。
7. 复核完成。

## 14. 角色与操作边界

### 14.1 管理员可执行操作

1. 查看默认 Taxonomy 模板。
2. 按三级叶子节点选择保留或不选择默认模板分类。
3. 输入业务提示词并生成企业特色 Taxonomy 草稿。
4. 预览并确认初始 Taxonomy 生效。
5. 发起批量打标任务。
6. 选择打标来源为原文或 Summary。
7. 选择是否启用 Taxonomy 优化。
8. 查看打标结果、优化结果和 Taxonomy 版本变化。
9. 人工新增、修改、移动、停用、删除节点。
10. 查看节点变更的级联影响。
11. 对删除、停用、重定义等高影响操作进行二次确认。
12. 从未打标签池、待复核池或待重新打标池中批量触发处理。

### 14.2 打标 Agent 操作边界

1. 可以基于任务绑定的 Taxonomy 版本输出三级叶子标签。
2. 可以识别 Taxonomy 无法覆盖的文档内容。
3. 可以在开启优化时输出候选新增三级叶子标签。
4. 不可以直接删除标签。
5. 不可以直接合并标签。
6. 不可以决定 Taxonomy 版本更新时间。

### 14.3 Taxonomy 健康自检 Agent 操作边界

1. 可以判断候选新增标签是否冗余。
2. 可以建议复用已有三级叶子标签。
3. 可以判断不冗余的候选三级叶子标签可新增。
4. 可以提示节点修改、移动、停用或删除的潜在风险。
5. 不可以删除已有标签。
6. 不可以合并已有标签。
7. 不可以决定 Taxonomy 版本更新时间。

### 14.4 普通使用者感知范围

1. 查看文档 Summary。
2. 查看文档标签结果。
3. 使用 Taxonomy 进行知识库筛选和检索。
4. 在权限允许范围内使用最终已生效的标签和摘要结果。
5. 不参与 Taxonomy 生成、优化和维护确认。

## 16. 开发交付判定口径

当以下结果均可被业务侧验证时，可认为本次核心能力范围已覆盖：

1. 批量导入的文本文件可自动生成 Summary，并可查看与编辑。
2. Summary 被人工修改后，文档可回归未打标签池或待重新打标池。
3. 系统可提供企业通用默认 Taxonomy 模板。
4. 管理员可按三级叶子节点选择保留或不选择默认模板分类。
5. 管理员可基于行业、企业描述或知识库内容生成企业特色 Taxonomy 草稿。
6. 管理员可预览、修改并确认一套三级 Taxonomy 生效。
7. 初始 Taxonomy 确认后可形成版本。
8. 管理员可选择文档、选择打标来源，并完成批量打标。
9. 批量打标任务可绑定 Taxonomy 版本。
10. 文档最终标签结果只绑定三级叶子标签。
11. 打标结果优先复用当前任务绑定的 Taxonomy 版本。
12. 管理员可选择是否在打标中启用 Taxonomy 优化。
13. 启用优化时，系统可输出候选新增三级叶子标签。
14. 候选新增标签可经过健康自检 Agent 判断是否冗余。
15. 冗余候选标签可复用已有三级叶子标签。
16. 非冗余候选标签可作为任务内新增标签自动应用到触发文档，并在任务完成后统一落入新的 Taxonomy 版本。
17. 管理员人工新增标签时，自检 Agent 可检查是否冗余。
18. 管理员修改、移动、停用或删除节点时，系统可展示影响分析。
19. 删除或停用已应用标签后，受影响文档可回归未打标签池、待复核池或待重新打标池。
20. 未打标签池和待重新打标池文档可自动加入待打标队列，或由管理员批量触发重新打标。
21. Taxonomy 三级叶子标签可被知识库检索、筛选和展示消费。
22. Query 自动命中 Taxonomy 叶子标签时可增强检索，但不会阻断原有检索。
23. Prompt 可通过业务样例人工打磨确认，并用于开发测试验收。
24. 系统可清晰展示任务进度、文档结果、候选标签、自检结果、Taxonomy 版本和变更影响。
